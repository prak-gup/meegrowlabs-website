The 60-second version

The claim: treat your model and coding agent as a fixed black box. Don't chase them. Instead improve the only two levers you actually control — context and tools — and curate the environment around them.

Why it works: general model weights contain the visible tip of an organisation's iceberg. Below the waterline sit your current operational state, your local vocabulary, your quality bar, your procedures, your exception history, your authority relationships. No model will intuit those. Harness engineering is the last-mile work of making them available to a capable worker.

"Most people do not know that they can just point their agents at my writing, tweets, podcasts, and talks and improve the output of their agents by 100x." — Ryan Lopopolo

Where you stand: you have built an enormous amount of harness. The problem is not that you're missing pieces — it's that you have too many undifferentiated pieces with no routing layer, no authority boundary, and no proof loop. In this framework's own language: your context store is world-class and your retrieval is broken.

Part one

What harness engineering actually is

The mental model, and why it's different from prompt engineering, context engineering, and agent frameworks.

The black box move

Most people optimising AI output reach for the model: better prompts, newer weights, a bigger context window, a fancier framework. Harness engineering deliberately freezes that. You pick a model and a coding agent, declare it constant for an "adoption epoch," and stop treating it as the variable.

What's left to improve? Two external levers and one surrounding practice:

LeverWhat it meansConcrete form
Context What the worker can recover about intent, state, constraints, and quality bar Routing files, repo structure, types, examples, runbooks, failing checks, mounted corpora
Tools What the worker can actually do to the real system, and whether it can tell what happened CLIs, MCP servers, domain commands, diagnostics that name the violated invariant
Environment Everything around those: authority, proof, feedback, lifecycle Permission contracts, real-system tests, review loops, maintenance schedules

The five-part promise

The README states the goal precisely. A well-harnessed worker should be able to:

  1. Recover intent — figure out the real outcome behind a sparse prompt
  2. Operate the real system — not a mock, not a description of it
  3. Respect authority — know what it may and may not consequentially change
  4. Prove the outcome — with evidence matched to the actual claim
  5. Leave the next run better equipped — the loop compounds

Read those as a diagnostic. When an agent session goes badly, exactly one of those five broke first. Finding which, and where it should have been owned, is the entire practice.

Why this is not "context engineering"

Context engineering is one lever of three. Harness engineering adds the parts people skip: the tools loop (discover → select → invoke → interpret → repair → verify), the authority contract (capability ≠ permission), the proof boundary (a green check proves only its own assertion), and the feedback machinery that turns yesterday's correction into today's constraint.

The one-line test

The repo states its own success criterion bluntly: "The bundle succeeds when it changes engineering decisions, not when an agent repeats its terminology." Apply the same bar to your own harness. A skill that never fires changed nothing.

How the repo is built (and why that itself is the lesson)

The repository is not a library. It is a retrieval-optimised context bundle — a worked example of its own thesis. Five layers, each pointing one level deeper without repeating the layer above:

README.md          → the practice, the promise, entry points
AGENTS.md          → routing: "unresolved decision X? read thesis Y"
ARCHITECTURE.md    → who owns which file; the routing invariants
docs/README.md     → index of the twelve developed arguments
docs/<thesis>/     → the argument, the cases, the applications
playbooks/         → procedures you can actually run
sources/           → preserved evidence + provenance

The critical design choice: the raw corpus is not part of the default working set. An agent loads AGENTS.md, reads your target system, names the one decision local evidence can't resolve, and pulls exactly one thesis. That's it. This is the thing your own setup is missing most.

Partial

Also: retire scaffolding the new worker has absorbed.

05

Make capabilities legible and operable

Partial

A tool is only usable if the worker can complete the whole loop: discover → select → invoke → interpret → repair → verify. Two separate axes: discoverability (does the worker know it exists?) and familiarity (does the worker already know how to use it?). A POSIX CLI has huge learned priors but is invisible until announced. An MCP server advertises itself but every call shape is novel.

Design every tool result as context: quiet success; on failure, name the violated invariant, the affected target, and a known recovery action. Never let the agent pipe an expensive test command through head — own that at the tool boundary instead.

Part four

The five findings that matter

Ranked by output gained per hour spent. Each names the earliest owner — the place a fix stops the problem recurring.

1

You have a library with no card catalogue

Highest impact

Evidence. 963 SKILL.md files totalling 937,695 lines in ~/.claude/skills. 391 top-level skills, 93 agents, 10 plugins. Your global CLAUDE.md is 838 bytes and explicitly routes to exactly two of them (graphify, orchestrate).

Why it hurts. Every one of those skills has to be surfaced by name-and- description matching against your prompt. With 391 candidates the signal-to-noise is terrible: near-duplicates compete, the right skill loses to a plausible neighbour, and you get "why didn't it use X?" You paid the build cost and aren't collecting the return.

Earliest owner. A real routing file — the AGENTS.md pattern. Not a list of skills; a list of unresolved decisions mapped to destinations. The harness repo's own AGENTS.md is the model: "Deciding whether the worker lacks a capability or whether an existing tool is hard to discover → read docs/tool-legibility/."

Also. Your second-brain is loaded via "read /Users/prakhar/second-brain/CLAUDE.md at the start of every session." That's deterministic stuffing. By turn 40, post-compaction, it's gone — and nothing tells the agent how to get it back. Replace with a route the agent can re-derive at any point.

2

Capability and authority are fully conflated

Highest risk

Evidence. "defaultMode": "auto", "skipDangerousModePermissionPrompt": true, "skipAutoPermissionPrompt": true, "deny": []. The allowlist includes Bash(bash) — arbitrary shell — plus Bash(sudo npm *), Bash(brew install *), Bash(cargo install *), Bash(npx:*).

Why it hurts. The thesis is maximize autonomy inside explicit authority, and you've implemented the first half only. Right now an agent working on a throwaway script has identical power to one touching client data. There is no signal distinguishing reversible from irreversible, and no recovery boundary.

Earliest owner. The deny list — mechanically enforced, never depends on the agent remembering. This is the cheapest fix in this document: about ten minutes. You are not giving up speed; a deny list of 6–8 genuinely irreversible operations leaves your broad envelope completely intact for the reversible 99%.

The specific exposure

You handle client work for SYNC, AIBS, and Meegrow, with live MCP write access to Asana, Notion, WordPress (publish_wordpress), social posting (post_social), and Windsor ad-platform mutations that can change real ad budgets. Those are consequential, externally-visible, and in several cases irreversible — currently reachable with no gate.

3

Your feedback loop has no review boundary

Root cause of #1

Evidence. claudeception-activator.sh fires on every UserPromptSubmit with "MANDATORY … NON-NEGOTIABLE … you MUST evaluate". The agent then self-assesses whether its own work was novel, and writes a skill into the live retrieval path if it decides yes.

Why it hurts. This is the mechanism that produced 963 files, and the harness repo warns about it in almost these words:

"Candidate material remains untrusted and excluded from the parent's retrieval path. It cannot trigger publication directly… Corroborate those self-reports before publishing them as context." — docs/just-in-time-context/

An agent grading its own novelty is not corroboration. Worse, the hook fires on trivial prompts too, spending attention on extraction that should have gone to the task.

Earliest owner. A staging directory outside the retrieval tree. Candidates land in ~/.claude/candidates/. You promote weekly, by hand, only what recurred. The instinct is right — lessons should become infrastructure — you just need the gate that makes the lesson durable instead of noisy.

4

No default proof path

Medium

Evidence. 2/7 repos have CI, 3/7 have any tests, 0/7 have ARCHITECTURE.md. Orchestra has a genuine test gate, but it only applies to orchestrated builds — the majority of your sessions bypass it.

Why it hurts. "Match evidence to the claim." Most of your output is client-visible: sites, ads, content, SEO deliverables. The claim is "this renders correctly and says the right thing to a real user" — and a passing build proves nothing about that. Your review attention is currently the proof mechanism, which is the most expensive possible option.

Earliest owner. A per-repo ARCHITECTURE.md naming the journey that must hold, plus one real check per repo — a Playwright journey, a link check, a schema validation. You already have Playwright installed and a Chrome MCP connected. The capability exists; the default path doesn't.

5

Nothing is measured, so nothing gets retired

Compounding

Evidence. No evals/ anywhere. No record of which skills fire. gstack-ship exists in 11 copies at 3056/3047/3045/2786/2781/2720/2669 lines — already diverged. gstack-upgrade: 11 copies. gstack: 10.

Why it hurts. Thesis 12: tokens, skills, and agents are inputs. Without measurement you can only ever add. That's precisely how a system reaches 937k lines of guidance and gets worse at retrieval — every addition dilutes the rest.

Earliest owner. A one-line-per-session record and a quarterly GC pass. The bar from the repo: "Remove it when it adds noise, duplicates a better owner, or repeatedly fails to improve the job." Removal is a first-class harness operation, not an admission of failure.

Theses 10 & 12 · Coherence, effectiveness

Part six

The loop to run forever

Once the five moves are done, this is the practice itself — the improve-harness playbook, adapted to how you work.

The repo's core operational loop, verbatim:

baseline → earliest gap → smallest owning intervention → native verification → fresh rerun → retain, revise, or remove

Run it on one real job — small enough to rerun, important enough to exercise the gap. Not "make agents better." Something like "produce a client SEO audit for AIBS."

StepWhat you doWhat you're looking for
1. BaselineRun the job in a fresh session. Don't help.Where did you have to intervene? Every relay is a harness gap.
2. Earliest failed handoffTrace the symptom upstream to the first missing thing.Classify: context / capability / domain ownership / authority / proof / feedback. Fix the earliest, not the loudest.
3. HypothesisWrite it before implementing: "If <X> is added at <owner>, the worker will <observable change> because <mechanism>."Also write what would weaken it, and its carrying cost.
4. Implement + verifySmallest reversible change at the earliest owner.Verify twice: internal checks, then the actual user journey.
5. Fresh rerunSame worker, same authority, brand new session.Confirm the intervention was actually retrieved. A win from something never used proves nothing.
6. DecideRetain / revise / remove.If unsure it helped: temporarily remove it and rerun. Test-without is the strongest cheap evidence you can get.
The discipline that makes it work

Step 5 is where almost everyone cheats. If you evaluate in the same session where you built the fix, the conversation itself is carrying the context — you're measuring the conversation, not the harness. Fresh session, or the result is worthless.

Two constraints the playbook is emphatic about, both of which are easy to violate when you're the one who built the thing:

  • Never weaken the grader to make a run pass. If the check is too strict, that's a separate decision made separately.
  • One failed trajectory is not a model limitation. Leave it as an open candidate until repeated evidence separates it from randomness and from environment gaps you haven't found yet.

Where to point it first

Run the loop on your highest-frequency job. For you that's almost certainly a client deliverable — an SEO audit, an ads plan, or a content piece. Highest frequency means the fastest feedback and the biggest compounding return, and it exercises the second-brain routing you just rebuilt in move 2.

What NOT to add

Don'tBecause

The honest summary

You've been building harness for a while without the vocabulary for it — the second-brain, orchestra, the skill system, the hooks are all harness engineering. What the framework gives you is not new components.
