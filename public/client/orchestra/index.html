01The four workers

Each slice goes to the cheapest engine that can do it well. Your flat-rate plans stay flat-rate — no per-token gateway.

WorkerEngineBest for
claudeClaude MaxArchitecture, hard logic, security — the premium seat, used sparingly.
glmGLM-5.2 (z.ai)The everyday workhorse. Strong + flat-rate; doesn't burn the Max seat.
codexCodexCode QA, review, refactor — a strict second set of eyes.
localQwen3-Coder · LM StudioTests, boilerplate, bulk, anything offline. Free, on-device.

02Routing is automatic — and editable

You describe the task; a version-controlled policy.yaml decides the route. Change a keyword or a chain there and every future run picks it up. No more hand-writing routes.

# policy.yaml — the rules, in one editable file
buckets:
  hard:   chain: claude,glm,local   qa: true    # architecture, security, migrations
  bulk:   chain: local              # typos, renames, docstrings, formatting
defaults: chain: glm,local                    # everyday work

03Reactive failover, built in

A route is a chain. If Claude hits its usage cap, the same slice auto-falls to GLM, then to free local — no prediction, no babysitting. The dashboard shows which engine actually ran.

claude→ capped? → glm→ down? → local

04Every slice is isolated & gated

Own worktree

Each slice runs on its own orch/<slug> branch in a separate git worktree — parallel agents never stomp each other.

Deps linked in

Your .venv / node_modules are symlinked into every worktree, so tests actually run on existing projects.

QA pass

--qa runs a Codex review + refactor over the output before the gate — fix bugs, tighten edges, add types.

You merge

Tests run, a PR is offered — but nothing merges into your branch without your eyes on the diff.

05The pipeline — plan, then execute

e2e-dev (Matt Pocock method) decomposes the work; the Orchestra executes it.

YOU ─ "build X" ─▶ ORCHESTRATOR (one session you talk to) │ /e2e-dev → grill → PRD → tracer-bullet issues (HITL / AFK) ▼ /orchestrate → fan AFK slices in parallel: ┌───────────────┬───────────────┬───────────────┐ claude,glm,local glm,local local +--qa (Codex) (hard) (everyday) (bulk/tests) └──── each in its own worktree → test gate → you review → merge ────┘ │ herdr panes (live, persistent, ssh-reattach) + `orch` dashboard

06Watch it work

A terminal dashboard reads the run ledger and live panes — status, which engine ran, whether it failed over, QA, tests, files, branch.

$ orch status
✓  add-auth      glm(claude,glm,local)⇢  qa:done  tests:pass   6  orch/add-auth
✓  list-endpoint glm(glm,local)        qa:none  tests:pass   3  orch/list-endpoint
✓  fix-typo      local(local)          qa:none  tests:skip   1  orch/fix-typo
   ⇢ = Claude was capped, GLM took over automatically

07Cost posture

Push work down the ladder as far as quality allows. Nothing routes through a paid API gateway — subscriptions stay flat-rate, and the bulk runs free on-device.
